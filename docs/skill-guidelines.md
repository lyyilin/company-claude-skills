## 📁 目录结构

### 必需结构

每个 Skill 必须遵循以下目录结构：

```
skills/your-skill-name/
├── SKILL.md              # 必需：技能说明文档
├── instructions.md       # 必需：技能指令
└── README.md            # 可选：额外说明
```

### 可选文件

根据需要，可以添加：

```
skills/your-skill-name/
├── SKILL.md
├── instructions.md
├── README.md
├── examples/            # 示例文件
│   └── example.txt
├── templates/           # 模板文件
│   └── template.md
└── assets/             # 资源文件
    └── diagram.png
```

## 📄 文件要求

### SKILL.md（必需）

这是 Skill 的主要说明文档，必须包含：

#### YAML Frontmatter（必需）

```yaml
---
name: skill-name              # 必需：技能名称（小写，连字符分隔）
description: 简短描述          # 必需：一句话说明功能
version: 1.0.0                # 必需：语义化版本号
author: 作者名                # 必需：创建者
created: 2026-01-13           # 必需：创建日期（YYYY-MM-DD）
updated: 2026-01-13           # 可选：最后更新日期
tags:                         # 可选：标签
  - pdf
  - analysis
dependencies:                 # 可选：依赖项
  - other-skill-name
---
```

#### 文档内容（必需）

```markdown
# Skill 名称

## 功能描述
详细说明这个 Skill 做什么，解决什么问题。

## 使用场景
列出适合使用这个 Skill 的场景。

## 使用方法
说明如何使用这个 Skill。

## 配置说明
如果有配置项，在这里详细说明。

## 示例
提供实际的使用示例。

## 限制和注意事项
说明已知的限制和使用注意事项。
```

### instructions.md（必需）

这是给 Claude 的指令文件：

```markdown
# 技能指令

## 目标
说明这个 Skill 的目标...

## 工作流程
1. 第一步...
2. 第二步...
3. 第三步...

## 具体指令
详细的指令内容...

## 输出格式
说明期望的输出格式...
```

**要求：**
- ✅ 指令清晰、具体
- ✅ 逻辑结构合理
- ✅ 包含必要的上下文
- ✅ 说明输出格式

### README.md（可选）

如果需要额外说明，可以添加 README.md：

```markdown
# Skill 名称

额外的说明、安装步骤、依赖说明等...
```

## 🏷️ 命名规范

### Skill 目录命名

**规则：**
- 使用小写字母
- 使用连字符（-）分隔单词
- 使用描述性名称
- 避免使用缩写
- 长度建议 2-4 个单词

**✅ 好的命名：**
```
pdf-analyzer
code-reviewer
meeting-summarizer
data-visualizer
```

**❌ 不好的命名：**
```
MySkill           # 使用了大写
skill1            # 不够描述性
pdfAnalyzer       # 使用了驼峰命名
pdf_analyzer      # 使用了下划线
pa                # 使用了缩写
```

### 文件命名

- `SKILL.md` - 全大写（固定名称）
- `instructions.md` - 全小写（固定名称）
- `README.md` - 全大写（固定名称）
- 其他文件使用小写和连字符

## 📝 文档标准

### 语言和风格

- ✅ 使用清晰、简洁的中文
- ✅ 使用主动语态
- ✅ 避免行话和缩写
- ✅ 提供具体的示例
- ✅ 使用 Markdown 格式化

### 文档结构

1. **标题层级**
   ```markdown
   # 一级标题（Skill 名称）
   ## 二级标题（主要章节）
   ### 三级标题（子章节）
   ```

2. **代码块**
   ````markdown
   ```语言
   代码内容
   ```
   ````

3. **列表**
   ```markdown
   - 无序列表项
   - 另一个项
   
   1. 有序列表项
   2. 另一个项
   ```

4. **强调**
   ```markdown
   **粗体** - 重要内容
   *斜体* - 强调
   `代码` - 代码或命令
   ```

### 示例质量

提供的示例必须：
- ✅ 真实可用
- ✅ 覆盖常见场景
- ✅ 包含输入和输出
- ✅ 说明预期结果

**示例格式：**

```markdown
## 示例

### 场景：分析 PDF 文档

**输入：**
```
请分析这个 PDF 文档的主要内容
```

**输出：**
```
文档分析结果：
1. 主题：...
2. 关键点：...
3. 结论：...
```
```

## 🎯 最佳实践

### 1. 单一职责

每个 Skill 应该专注于一个明确的任务：

**✅ 好的设计：**
- `pdf-text-extractor` - 只提取 PDF 文本
- `pdf-summarizer` - 只总结 PDF 内容

**❌ 不好的设计：**
- `pdf-handler` - 做太多事情（提取、总结、转换等）

### 2. 清晰的接口

使用清晰的触发词和参数：

**✅ 好的接口：**
```markdown
触发词：分析 PDF
参数：文件路径、分析深度
```

**❌ 不好的接口：**
```markdown
触发词：处理
参数：不明确
```

### 3. 完整的文档

提供所有必要信息：

**必须包含：**
- 功能描述
- 使用场景
- 使用方法
- 配置说明
- 示例
- 限制说明

### 4. 版本管理

使用语义化版本号：

```
主版本号.次版本号.修订号

1.0.0 - 初始版本
1.1.0 - 添加新功能
1.1.1 - 修复 Bug
2.0.0 - 重大更新（不兼容旧版本）
```

### 5. 依赖管理

如果 Skill 依赖其他 Skill 或外部工具：

```yaml
---
dependencies:
  - other-skill-name
  - external-tool-name
---
```

在文档中说明：
```markdown
## 依赖项

本 Skill 需要：
- `other-skill-name` - 用于...
- `external-tool-name` - 用于...

安装方法：...
```

### 6. 测试

在提交前测试你的 Skill：

- ✅ 在多个场景下测试
- ✅ 测试边界情况
- ✅ 验证文档准确性
- ✅ 检查示例可用性

### 7. 持续改进

根据使用反馈改进 Skill：

- 收集用户反馈
- 修复发现的问题
- 添加新功能
- 更新文档
